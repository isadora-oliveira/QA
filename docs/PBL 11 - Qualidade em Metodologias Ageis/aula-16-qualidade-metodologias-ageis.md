# 1. Análise de Práticas Ágeis no Processo

O projeto de Qaulidade do **LocalEats** foi desenvolvido individualmente. Algumas práticas ágeis foram utilizadas de forma simples durante o desenvolvimento, enquanto outras não foram aplicadas por não haver uma equipe trabalhando no projeto.

| Prática | Existe no processo? | Como é aplicada atualmente? | Pode ser melhorada? |
|----------|---------------------|-----------------------------|---------------------|
| Planejamento iterativo | Parcialmente | O desenvolvimento foi feito por etapas, conforme novas funcionalidades eram necessárias. | Sim. Organizar o projeto em pequenas etapas com objetivos definidos. |
| Priorização de funcionalidades | Sim | Primeiro foram desenvolvidas as funcionalidades mais importantes para o sistema funcionar. | Sim. Criar uma lista organizada de prioridades antes de começar cada etapa. |
| Entregas incrementais | Sim | As funcionalidades foram sendo adicionadas aos poucos durante o desenvolvimento. | Sim. Fazer entregas menores e testar cada uma antes de continuar. |
| Feedback frequente | Parcialmente | O feedback aconteceu principalmente durante os testes feitos por mim | Sim. Buscar mais opiniões do professor e dos colegas durante o desenvolvimento. |
| Trabalho colaborativo | Não | O projeto foi desenvolvido por apenas um aluno. | Sim. Em um projeto em grupo seria possível dividir tarefas e trocar ideias com os colegas. |
| Controle visual das atividades | Não | Não foi utilizado Kanban ou outra ferramenta para organizar as tarefas. | Sim. Utilizar Trello ou GitHub Projects para acompanhar o andamento do projeto. |
| Melhoria contínua | Sim | Durante o desenvolvimento foram feitas correções e pequenas melhorias sempre que necessário. | Sim. Reservar momentos para revisar o projeto e identificar melhorias. |

## Conclusão

O projeto utilizou algumas práticas que fazem parte das metodologias ágeis, como desenvolver o sistema aos poucos e priorizar as funcionalidades mais importantes. Como o trabalho foi feito individualmente, não foi possível aplicar práticas como reuniões e trabalho em equipe. Mesmo assim, o processo pode ser melhorado com uma organização maior das tarefas, planejamento por etapas e definição de critérios para iniciar e finalizar cada funcionalidade.

# 2. Propostas de Melhoria Ágil

| Melhoria Proposta | Metodologia Relacionada | Benefício Esperado |
|-------------------|------------------------|--------------------|
| Utilizar um quadro Kanban para organizar as tarefas. | Kanban | Facilitar o acompanhamento do que já foi feito e do que ainda falta. |
| Planejar o desenvolvimento em pequenas etapas (sprints). | Scrum | Melhor organização e controle das entregas. |
| Criar uma Definition of Ready antes de iniciar uma funcionalidade. | Scrum | Evitar começar uma tarefa com dúvidas ou informações incompletas. |
| Criar uma Definition of Done para todas as funcionalidades. | Scrum | Garantir que todas sejam entregues com o mesmo padrão de qualidade. |
| Fazer mais testes durante o desenvolvimento. | XP | Encontrar erros mais cedo e evitar retrabalho. |
| Reservar um momento para revisar o processo ao final de cada etapa. | Lean / Scrum | Identificar o que pode ser melhorado nos próximos desenvolvimentos. |

---

# 3. Definition of Ready (DoR)

Uma funcionalidade estará pronta para começar o desenvolvimento quando:

- O objetivo da funcionalidade estiver claro.
- Os requisitos estiverem definidos.
- Os critérios de aceitação estiverem descritos.
- Não existirem dúvidas sobre o que deve ser desenvolvido.
- A prioridade da funcionalidade estiver definida.

---

# 4. Definition of Done (DoD)

Uma funcionalidade será considerada concluída quando:

- Todos os requisitos forem atendidos.
- A funcionalidade funcionar corretamente.
- Os testes forem realizados sem apresentar erros.
- O código estiver organizado.
- A funcionalidade estiver integrada ao restante do sistema.
- A documentação necessária estiver atualizada, quando houver.
